## Remaining Work

### Priority 1 — Blocking / security

**WebSocket endpoints have no authentication**
File: `src/services/cloud/api-gateway/src/main.rs`

`/ws`, `/ws/raspi`, and `/ws/packets` bypass the API key middleware. Any client can connect and — via `/ws/raspi` — inject block/unblock commands to the Pi.

Fix: extract the key check into a helper, call it at the top of each WS handler before the upgrade. Accept key via `?api_key=` query param for browser dashboard clients.

---

### Priority 2 — Important, not yet blocking

**Add packet-processor and rule-engine to docker-compose.raspi.yml**
Files: `docker-compose.raspi.yml`, `src/services/edge/packet-processor/Dockerfile`, `src/services/edge/rule-engine/Dockerfile`

Both services are code-complete. Without them:
- Raw packet capture to VPS via `/ws/packets` never happens
- Rules generated by the VPS are not applied on the Pi via rule-engine

Also fix: rule-engine has a hardcoded `suricatasc` socket path — make it configurable via `SURICATA_SOCKET_PATH` env var.

**Rule Expiry / Unblock**
Files: `src/services/edge/packet-processor/src/main.rs`, `src/services/edge/network-filter/src/main.rs`

Currently `block_command` adds a blocked IP permanently. `expires_at_dt` is stored in MongoDB but nothing reads it.

Add:
- Background task in api-gateway (every 60 s): query expired active records, set `active = false`, broadcast `unblock_command` via `/ws/raspi`
- `unblock_command` handler in raspi-collector → calls network-filter `/api/v1/unblock`
- TTL eviction in packet-processor's blocked_ips DashMap

**Rate-limit `/ws/packets`**
File: `src/services/cloud/api-gateway/src/main.rs`

`MAX_PACKETS_PER_SECOND` and `RATE_LIMITING_ENABLED` are defined but not enforced on the WS path.

Add:
- Per-connection token-bucket counter in the WS handler loop
- Drop frames (or close connection) when limit is exceeded

**threat-intel service**
File: `src/services/cloud/threat-intel/src/main.rs`

TCP stub — opens port 8092 and drops every connection. The api-gateway `/api/threat-intel` endpoint works around this with a direct MongoDB aggregate but has no external feed data.

Implement:
- IP/domain reputation lookups from internal MongoDB collection
- Background updater pulling from a public feed (e.g. Abuse.ch Feodo Tracker)
- HTTP API: `GET /threat-intel` → `{ malicious_ips, suspicious_domains, vulnerabilities }`

### Priority 3 — Nice to have / operational improvements

**Telemetry VPS streaming**
File: `src/services/edge/telemetry/src/main.rs`

Collects CPU/memory/disk/temp metrics locally, but does not send them to the VPS. The dashboard and Grafana receive no Pi hardware metrics.

Add a background loop that POSTs to `/api/telemetry` on the VPS every `COLLECTION_INTERVAL_SECS`. Include `X-API-Key` header.

**ids-pi IDS logic**
File: `src/services/edge/ids-pi/app.py`

58 lines — only responds to `/health` and `/`. No nuclei scanning or scheduled security checks.

Either implement nuclei scan scheduler and POST results to the VPS, or remove the service and its entry in docs if Suricata covers this use case.

**CIDR whitelist code duplication**
File: `src/services/cloud/api-gateway/src/main.rs`

Inline `cidr_contains()` duplicates `idps_utils::network_utils::is_in_cidr()` which is already complete and tested. Refactor to use the shared library.

### Priority 4 — Hardening & Production-readiness

**Suricata Alert Feedback Loop (VPS side)**
The `src/services/cloud/log-processor/` directory has a Dockerfile stub but no source code. Without it, Suricata alerts from the VPS `packet-analyzer` are never ingested into MongoDB or shown in the dashboard.

Options:
- Implement as a small Rust service (tail + `POST /api/alerts/ingest`)
- Use a Filebeat/Vector sidecar calling the ingest endpoint
- If VPS Suricata is not needed, delete the stub and remove references from compose

**Integration / End-to-end Tests**
No automated tests cover the Pi→VPS event path.

Add:
- Integration test: spin up api-gateway + mock Pi collector, POST a crafted event, assert it appears in MongoDB
- Unit tests for rule-engine parser (valid + malformed Suricata rule strings)
- Unit tests for packet-processor blocked_ips matcher

**WireGuard Key Rotation Runbook**
WireGuard keys are static. If the Pi is compromised there is no documented procedure for rotation. Add a section to OPERATIONS.md covering:
- Generate new keypair on Pi
- `wg set wg0 peer <old-key> remove` on VPS
- Add new peer on VPS with `wg set wg0 peer <new-key> allowed-ips 10.10.0.2/32`
- Update `.env` on Pi with new `WG_PRIVATE_KEY`; restart WireGuard container

## Known Limitations

| Item | Note |
|---|---|
| `packet-processor` / `rule-engine` not in compose | Full packet capture and dynamic rule enforcement not yet active |
| WebSocket endpoints unauthenticated | `/ws`, `/ws/raspi`, `/ws/packets` — any client can connect |
| Blocked IPs never expire | `expires_at_dt` stored but no eviction background task |
| `threat-intel` stub | All threat intelligence data comes from MongoDB aggregates only |
| `ids-pi` minimal | No IDS logic — just a health endpoint |
| Telemetry not streaming | Pi hardware metrics not visible in Grafana/VPS dashboard |
| `rule-engine` socket path hardcoded | `/var/run/suricata/suricata-command.socket` — should be `SURICATA_SOCKET_PATH` env var |
| `raspi-collector` reconnect backoff | Fixed 5 s wait on VPS disconnect — should use exponential backoff with jitter |
